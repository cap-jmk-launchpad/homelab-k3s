# GitLab CE on homelab k3s

Self-hosted **GitLab Community Edition** (Omnibus container) plus **GitLab Runner** with the **kubernetes** executor. Lives in namespace **`gitlab`** — separate from **`supabase`** and other stacks.

## Quick checklist (first-time setup)

1. **Secrets + `.env`** (launchpad, not committed):
   ```bash
   LAUNCHPAD_ENV=~/launchpad/.env ./scripts/k8s-gitlab-secret.sh
   ```
   Reuses existing `GITLAB_*` keys unless `GITLAB_REGENERATE_SECRETS=1`.

2. **Deploy** (local kubeconfig or remote to blackpearl):
   ```bash
   ./scripts/k8s-gitlab-apply.sh
   # or: GITLAB_REMOTE=1 STAGING_HOST=blackpearl LAUNCHPAD_ENV=~/launchpad/.env ./scripts/k8s-gitlab-apply.sh
   ```
   First Omnibus boot often takes **10–20 minutes**.

3. **Login** — user **`root`**, password from launchpad `.env` (`GITLAB_ROOT_PASSWORD`). Change it under **Admin → Account** if you have not already.

4. **Structure for klaut.pro products** (UI or API):
   - Create top-level group(s), e.g. **`klaut`** or per-product groups.
   - Create projects under those groups (mirror naming from [klaut-pro-products.md](klaut-pro-products.md) if helpful).
   - Enable **push mirror** or periodic `git fetch` to GitHub for code redundancy (see [Backup strategy](#backup-strategy)).

5. **Optional public URL** — `https://gitlab.klaut.pro` (see [Public edge](#public-edge-optional)).

6. **Backups** — apply stack (includes weekly CronJob), then smoke-test:
   ```bash
   ./scripts/k8s-gitlab-backup.sh
   ./scripts/k8s-gitlab-restore.sh   # list only
   ```

## RAM and placement

| Component | requests | limits | Notes |
|-----------|----------|--------|-------|
| GitLab CE (Omnibus) | 3 Gi / 500m CPU | 6 Gi / 2 CPU | GitLab docs recommend **≥4 Gi** for small installs; 6 Gi cap avoids OOM during upgrades |
| GitLab Runner | 128 Mi / 100m | 512 Mi / 500m | Manager only; CI job pods add their own requests |
| CI job pods (default) | runner config | per job | Tune in runner `config.toml` after register |

**Node:** manifests pin **`kubernetes.io/hostname: engine`** so the control plane (blackpearl) stays lighter — same idea as [homelab-move-workloads-to-engine.sh](../scripts/homelab-move-workloads-to-engine.sh). If `engine` is down, remove or change the `nodeSelector` in `k8s/gitlab/statefulset.yaml` and `runner-deployment.yaml`.

Bundled Omnibus monitoring (Prometheus/Grafana/exporters) is **disabled** in `k8s/gitlab/configmap.yaml` to save RAM. Container **registry is disabled** (`registry['enable'] = false`).

## Access

| Mode | URL |
|------|-----|
| NodePort (LAN) | `http://192.168.10.33:30481/` |
| Public (edge) | `https://gitlab.klaut.pro` (when edge route enabled) |
| In-cluster (runner) | `http://gitlab.gitlab.svc.cluster.local` |
| Port-forward | `kubectl -n gitlab port-forward svc/gitlab 8080:80` → `http://127.0.0.1:8080/` |

Git over SSH (optional): NodePort **30222** — requires Fritz/TCP forward or LAN-only use; HTTP(S) clone works without SSH.

## Public edge (optional)

Suggested hostname: **`gitlab.klaut.pro`**.

1. Set in launchpad `.env`:
   - `GITLAB_PUBLIC_URL=https://gitlab.klaut.pro`
   - `GITLAB_EXTERNAL_URL=https://gitlab.klaut.pro` (then re-run `k8s-gitlab-secret.sh` and restart GitLab)
2. Uncomment the `gitlab` upstream + `[[site]]` block in [k8s/edge/homelab.httpd.toml](../k8s/edge/homelab.httpd.toml) (or use [k8s/edge/Caddyfile](../k8s/edge/Caddyfile) if that is your edge).
3. DNS: **A** record `gitlab.klaut.pro` → WAN IP (Fritz forwards **443** → `192.168.10.33`, same as [search.klaut.pro](search-klaut-pro.md)).
4. On blackpearl: rsync edge config, `bash scripts/edge-lis-apply.sh`.

Until then, keep **`GITLAB_EXTERNAL_URL=http://127.0.0.1:30481`** for loopback/NodePort (proto-friendly).

## Credentials (launchpad `.env`)

Generated by `scripts/k8s-gitlab-secret.sh` (reused unless `GITLAB_REGENERATE_SECRETS=1`):

| Variable | Purpose |
|----------|---------|
| `GITLAB_NAMESPACE` | k8s namespace (`gitlab`) |
| `GITLAB_NODEPORT` | HTTP NodePort (`30481`) |
| `GITLAB_EXTERNAL_URL` | Omnibus `external_url` — **must stay stable** across redeploys |
| `GITLAB_URL` | Runner → GitLab API base |
| `GITLAB_PUBLIC_URL` | Optional public URL (edge) |
| `GITLAB_ROOT_PASSWORD` | `root` account |
| `GITLAB_RUNNER_REGISTRATION_TOKEN` | Instance runner registration (Omnibus initial token) |
| `GITLAB_REGISTRY_ENABLED` | `0` — registry off by default |
| `GITLAB_BACKUP_SKIP` | Optional; default skips registry/artifacts/builds/lfs/packages/terraform_state |

Never commit `.env`.

## Deploy and safe redeploy (no data loss)

### First deploy

```bash
LAUNCHPAD_ENV=../.env ./scripts/k8s-gitlab-secret.sh
./scripts/k8s-gitlab-apply.sh
```

Remote (rsync + apply on blackpearl):

```bash
GITLAB_REMOTE=1 \
  STAGING_HOST=blackpearl \
  STAGING_KEY=/path/to/blackpearl \
  LAUNCHPAD_ENV=../.env \
  ./scripts/k8s-gitlab-apply.sh
```

### Redeploy without losing data

| Do | Don't |
|----|-------|
| `kubectl apply -k k8s/gitlab/` or `./scripts/k8s-gitlab-apply.sh` | `kubectl delete pvc gitlab-data` |
| Re-run `k8s-gitlab-secret.sh` **without** `GITLAB_REGENERATE_SECRETS=1` | Regenerate secrets unless you intend a fresh GitLab |
| Keep the same `GITLAB_EXTERNAL_URL` in `.env` | Change `external_url` casually (breaks clone URLs, OAuth, etc.) |
| Upgrade image tag in `statefulset.yaml` and roll | Delete namespace `gitlab` wholesale |

**Persistent volumes (local-path on engine):**

| PVC | Size | Holds |
|-----|------|--------|
| `gitlab-data` | 50 Gi | Repos, Postgres, uploads, Omnibus backup staging (`/var/opt/gitlab/backups/`) |
| `gitlab-runner-config` | 1 Gi | Runner registration/config |
| `gitlab-backups` | 20 Gi | Weekly tarball copies for DR |

Omnibus **config** and **logs** use `emptyDir` — they are re-rendered from `gitlab-secrets` + `gitlab-config` on each pod start. All user/repo state lives on **`gitlab-data`**.

If GitLab was reinstalled but you kept `gitlab-data`, reuse the same `.env` credentials. If you regenerated secrets on existing data, runner registration may break — delete PVC `gitlab-runner-config` and restart the runner deployment.

## GitLab Runner (kubernetes executor)

- Registered automatically on runner pod start using `GITLAB_RUNNER_REGISTRATION_TOKEN`.
- Spawns CI pods in namespace **`gitlab`** with ServiceAccount **`gitlab-runner`** (RBAC in `runner-rbac.yaml`).
- Tags: **`k8s`**, **`homelab`**; runs untagged jobs.

Example `.gitlab-ci.yml` snippet:

```yaml
tags: [k8s]
image: alpine:3.20
script:
  - echo "hello from cluster"
```

If registration fails after GitLab was reinstalled, delete PVC `gitlab-runner-config` and restart the runner deployment.

## Backup strategy

Goal: protect **repository code and GitLab metadata** (issues, MRs, users, groups). CI images and container registry are **not** important on this homelab install.

### What `gitlab-backup create` includes

Always backed up:

- **Git repositories** (all projects)
- **PostgreSQL database** (users, groups, issues, MRs, permissions, CI config metadata, etc.)
- **uploads / attachments** (unless skipped)
- **repositories** bare and wiki repos

Skipped by default (`GITLAB_BACKUP_SKIP=registry,artifacts,builds,lfs,packages,terraform_state`):

| SKIP component | Why skip on homelab |
|----------------|---------------------|
| `registry` | Registry disabled in config; no container images stored |
| `artifacts` | CI job output blobs — reproducible from repo |
| `builds` | CI job logs |
| `lfs` | Skip if you do not use Git LFS; remove from SKIP if you do |
| `packages` | Package registry |
| `terraform_state` | Terraform state backend |

To include everything: unset `GITLAB_BACKUP_SKIP` or pass `SKIP=` empty (larger, slower backups).

### Automated backups (recommended)

- **Weekly** CronJob `gitlab-backup` — Sunday **04:00 UTC** — runs Omnibus backup, copies tarball to PVC **`gitlab-backups`**
- **Daily** prune CronJob — deletes files older than **30 days** on `gitlab-backups`
- Applied via `kubectl apply -k k8s/gitlab/`

### On-demand backup

```bash
./scripts/k8s-gitlab-backup.sh
```

### Restore

```bash
# List timestamps (prefix before _gitlab_backup.tar)
./scripts/k8s-gitlab-restore.sh

# Restore (stops puma/sidekiq briefly, scales runner down)
BACKUP_TS=20260101T120000Z ./scripts/k8s-gitlab-restore.sh
```

Off-cluster copy (optional):

```bash
POD=$(kubectl -n gitlab get pod -l app=gitlab -o jsonpath='{.items[0].metadata.name}')
kubectl -n gitlab cp "$POD:/var/opt/gitlab/backups/20260101T120000Z_gitlab_backup.tar" ./
# or from gitlab-backups PVC via a one-off alpine pod on engine
```

### Secondary: git mirror to GitHub

GitLab Omnibus backup is the **primary** DR for issues/MRs/users. For **code-only** redundancy:

1. **Per project** — **Settings → Repository → Mirroring repositories** → push mirror to GitHub.
2. **Bulk** — on a workstation with access to both remotes:
   ```bash
   git clone --mirror https://gitlab.klaut.pro/group/project.git
   cd project.git
   git push --mirror git@github.com:org/project.git
   ```
3. **Group export** — GitLab UI group export (includes repos + metadata, separate from Omnibus backup).

Mirrors do **not** replace Omnibus backup if you care about issues, MRs, or permissions.

### What NOT to backup

- Container **images** / registry (disabled)
- **CI runner cache** (ephemeral job pods)
- **Omnibus logs** on emptyDir (recreated)
- **Rendered** `/etc/gitlab/gitlab.rb` in emptyDir (recreated from secrets)

## Conflict with Supabase

Both can run concurrently: **different namespaces** (`supabase` vs `gitlab`), different NodePorts (**30480** vs **30481**). Watch **total RAM** on `engine` before leaving both up.

## Troubleshooting

```bash
kubectl -n gitlab get pods,pvc,events
kubectl -n gitlab logs statefulset/gitlab -f
kubectl -n gitlab logs deployment/gitlab-runner -f
kubectl top pod -n gitlab
kubectl -n gitlab exec statefulset/gitlab -- gitlab-rake gitlab:env:info
```

- **Pending pod on engine:** node missing label or insufficient memory → relax `nodeSelector` or add RAM.
- **502 / not ready:** wait for Omnibus reconfigure; check `kubectl logs`.
- **Liveness restarts but UI works:** probe hits `/-/health` before nginx routes it; usually settles once Ready; check `curl -s -o /dev/null -w '%{http_code}' http://127.0.0.1:30481/users/sign_in`.
- **Runner not picking jobs:** verify runner in **Admin → CI/CD → Runners**; check registration token matches `.env`.
