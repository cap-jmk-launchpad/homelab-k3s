# GitLab CE on homelab k3s

Self-hosted **GitLab Community Edition** (Omnibus container) plus **GitLab Runner** with the **kubernetes** executor. Lives in namespace **`gitlab`** — separate from **`supabase`** and other stacks.

## RAM and placement

| Component | requests | limits | Notes |
|-----------|----------|--------|-------|
| GitLab CE (Omnibus) | 3 Gi / 500m CPU | 6 Gi / 2 CPU | GitLab docs recommend **≥4 Gi** for small installs; 6 Gi cap avoids OOM during upgrades |
| GitLab Runner | 128 Mi / 100m | 512 Mi / 500m | Manager only; CI job pods add their own requests |
| CI job pods (default) | runner config | per job | Tune in runner `config.toml` after register |

**Node:** manifests pin **`kubernetes.io/hostname: engine`** so the control plane (blackpearl) stays lighter — same idea as [homelab-move-workloads-to-engine.sh](../scripts/homelab-move-workloads-to-engine.sh). If `engine` is down, remove or change the `nodeSelector` in `k8s/gitlab/statefulset.yaml` and `runner-deployment.yaml`.

Bundled Omnibus monitoring (Prometheus/Grafana/exporters) is **disabled** in `k8s/gitlab/configmap.yaml` to save RAM.

## Access (default: internal proto)

| Mode | URL |
|------|-----|
| NodePort (LAN) | `http://192.168.10.33:30481/` |
| In-cluster (runner) | `http://gitlab.gitlab.svc.cluster.local` |
| Port-forward | `kubectl -n gitlab port-forward svc/gitlab 8080:80` → `http://127.0.0.1:8080/` |

Login: **`root`** / password in launchpad `.env` (`GITLAB_ROOT_PASSWORD`).

Git over SSH (optional): NodePort **30222** — requires Fritz/TCP forward or LAN-only use; HTTP(S) clone works without SSH.

## Public edge (optional)

Live hostnames (same Omnibus pod, NodePort **30481**):

| Hostname | Docs / deploy |
|----------|----------------|
| `gitlab.klaut.pro` | [fritz-klaut-pro-port-forward.md](fritz-klaut-pro-port-forward.md) |
| `gitlab.d3bu7.com` | [beelink-cleanup/docs/gitlab-d3bu7-setup.md](https://github.com/cap-jmk-launchpad/beelink-cleanup/blob/master/docs/gitlab-d3bu7-setup.md) |
| `gitlab.lilangverse.xyz` | [beelink-cleanup/docs/gitlab-lilangverse-setup.md](https://github.com/cap-jmk-launchpad/beelink-cleanup/blob/master/docs/gitlab-lilangverse-setup.md) |

1. Set in launchpad `.env` (pick **one** canonical URL):
   - `GITLAB_PUBLIC_URL=https://<hostname>`
   - `GITLAB_EXTERNAL_URL=https://<hostname>` (then re-run `k8s-gitlab-secret.sh` and restart GitLab)
2. Ensure `[[site]] host = "<hostname>"` → `proxy:gitlab` in [k8s/edge/homelab.httpd.toml](../k8s/edge/homelab.httpd.toml).
3. DNS: **A** record → WAN `77.23.124.82` (Fritz forwards **80+443** → `192.168.10.33`).
4. On blackpearl: rsync edge config, `bash scripts/edge-lis-apply.sh` (include hostname in `HOMELAB_ACME_DOMAINS`).

Until a public hostname is chosen, keep **`GITLAB_EXTERNAL_URL=http://127.0.0.1:30481`** for loopback/NodePort (proto-friendly).

**Do not** deploy a second GitLab (e.g. Helm `gitlab-lilangverse` in `lis-work`) on engine without a migration plan — RAM and PVC conflict with this Omnibus install.

## Credentials (launchpad `.env`)

Generated by `scripts/k8s-gitlab-secret.sh` (reused unless `GITLAB_REGENERATE_SECRETS=1`):

| Variable | Purpose |
|----------|---------|
| `GITLAB_NAMESPACE` | k8s namespace (`gitlab`) |
| `GITLAB_NODEPORT` | HTTP NodePort (`30481`) |
| `GITLAB_EXTERNAL_URL` | Omnibus `external_url` |
| `GITLAB_URL` | Runner → GitLab API base |
| `GITLAB_PUBLIC_URL` | Optional public URL (edge) |
| `GITLAB_ROOT_PASSWORD` | `root` account |
| `GITLAB_RUNNER_REGISTRATION_TOKEN` | Instance runner registration (Omnibus initial token) |
| `GITLAB_REGISTRY_ENABLED` | `0` — registry off by default |

Never commit `.env`.

## Deploy

From homelab-k3s (or Windows with kubectl context to cluster):

```bash
LAUNCHPAD_ENV=../.env ./scripts/k8s-gitlab-secret.sh
./scripts/k8s-gitlab-apply.sh
```

Remote (rsync + apply on blackpearl):

```bash
GITLAB_REMOTE=1 \
  STAGING_HOST=blackpearl \
  STAGING_KEY=/path/to/blackpearl \
  LAUNCHPAD_ENV=../.env \
  ./scripts/k8s-gitlab-apply.sh
```

First Omnibus boot often takes **10–20 minutes**.

## GitLab Runner (kubernetes executor)

- Registered automatically on runner pod start using `GITLAB_RUNNER_REGISTRATION_TOKEN`.
- Spawns CI pods in namespace **`gitlab`** with ServiceAccount **`gitlab-runner`** (RBAC in `runner-rbac.yaml`).
- Tags: **`k8s`**, **`homelab`**; runs untagged jobs.

Example `.gitlab-ci.yml` snippet:

```yaml
tags: [k8s]
image: alpine:3.20
script:
  - echo "hello from cluster"
```

If registration fails after GitLab was reinstalled, delete PVC `gitlab-runner-config` and restart the runner deployment.

## Backup

```bash
./scripts/k8s-gitlab-backup.sh
```

Runs `gitlab-backup create` inside the Omnibus pod; artifacts live on PVC **`gitlab-data`**. Copy tarballs off-cluster for disaster recovery.

## Conflict with Supabase

Both can run concurrently: **different namespaces** (`supabase` vs `gitlab`), different NodePorts (**30480** vs **30481**). Watch **total RAM** on `engine` before leaving both up.

## Troubleshooting

```bash
kubectl -n gitlab get pods,events
kubectl -n gitlab logs statefulset/gitlab -f
kubectl -n gitlab logs deployment/gitlab-runner -f
kubectl top pod -n gitlab
```

- **Pending pod on engine:** node missing label or insufficient memory → relax `nodeSelector` or add RAM.
- **502 / not ready:** wait for Omnibus reconfigure; check `kubectl logs`.
- **Pod restart loop (liveness 404):** after changing `external_url` to a public HTTPS hostname, kubelet probes must send the matching **`Host`** header and use **`/users/sign_in`** (not `/-/health`, which 404s on pod IP). A **`startupProbe`** with `failureThreshold: 40` (~20 min) prevents liveness kills during Omnibus reconfigure. See `k8s/gitlab/statefulset.yaml`.
- **Runner not picking jobs:** verify runner in **Admin → CI/CD → Runners**; check registration token matches `.env`.
